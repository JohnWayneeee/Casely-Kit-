# Default Rules - Базовые правила написания тест-кейсов (fallback)

Используются, если `config/user-rules.mdc` ещё не создан после `/casely.init`.

## Универсальная структура
### Минимальные поля
- **ID**: уникальный идентификатор (формат `TC-001`, инкремент).
- **Название**: краткое описание проверки.
- **Предусловия**: начальное состояние системы (нумерованный список).
- **Шаги**: пронумерованные действия (императив).
- **Ожидаемый результат**: проверяемые результаты (маркированный список `-`).
- **Приоритет**: `Высокий` / `Средний` / `Низкий`.
- **Автор**: ФИО из параметра `/casely.init`.

### Базовый формат (Markdown)
```
## Тест-кейс: [Название]

**ID**: TC-001  
**Приоритет**: Высокий  
**Автор**: [Фамилия И.О.]

### Предусловия
1. [Условие 1]
2. [Условие 2]

### Шаги
1. [Шаг 1]
2. [Шаг 2]

### Ожидаемый результат
- [Результат 1]
- [Результат 2]
```

## Стиль
- Глаголы в шагах — императив: «Нажать», «Ввести», «Проверить».
- Нумерация шагов: `1.`, `2.`, `3.`.
- Ожидаемые результаты — маркированный список `-`.
- Предусловия — нумерованный список.

## Генерация ID
- Формат по умолчанию: `TC-###` с инкрементом.
- При наличии `user-rules.mdc` используем его паттерн.

## Метаданные
- Приоритет: `Высокий` / `Средний` / `Низкий`.
- Автор: из параметра `/casely.init`.
- Статус по умолчанию не задаётся (может быть добавлен в user-rules).

## Поддерживаемые форматы (MVP)
- Текст: `md`, `txt`.
- Изображения требований: `png`, `jpg` (OCR, точность зависит от качества).
- `docx/pdf` — не поддержаны в MVP; попроси конвертировать при попытке использования.

## Когда обновлять
- Эти правила — только fallback. После `/casely.init` всегда использовать `user-rules.mdc`.
